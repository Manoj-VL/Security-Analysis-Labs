# Objective
To  analyze real network traffic  using Wireshark and observe how the OSI layer works operate under DNS Resolution ,HTTP,HHTPS, TCP communications

## Setup
1. Downloaded Wireshark from Wireshark.org
2. Used my own machine's traffic to analyze the packets

## Steps Used
### 1. Generate Traffic
1. Started up the Wireshark and tapped the blue fin to start the packet capture
2. Visited the `google.com ,http://www.example.com,https://www.example.com `to generate traffic for packet capture
3. Stopped the packet capture by clicking the red button and started the packet analysis

### 2. Analysis 
#### Filter 1: DNS 
1. In the Filter column used DNS to Filter out DNS query and responses
2. Selected ```
```
   1.packet 24 Standard query 0x7c50 A www.google.com 
   2.packet 26 Standard query response 0x7c50 A www.google.com 
```
##### Packet 1 - DNS Query
-**Transaction ID** : 0x7c50, - randomly generated by the PC to match the query response
-**Record Type** **A** - requesting IPV4 address from `wwww.google.com`

-My PC (`192.168.1.15`) asks Router (`192.168.1.1`)
-Layer 4 UDP Source Port 61593 →Destination Port **53**
-Layer 7 DNS Query for `www.google.com`

##### Packet 2 -DNS  Response
-- Router (`192.168.1.1`) replies to your PC (`192.168.1.15`)
- Layer 4: UDP, Src Port **53** → Dst Port 61593
- Layer 7: DNS response containing the **IP address of google.com**

## Observation
*"The router acts as DNS Forwarder.. My PC sends an UDP request to port 53 on the router and router resolves domain upstream and returns the answer. Both packets demonstrates the encapsulation across layer 2,3,4,7 "*

#### Filter 2: HTTP/TCP
1. Generated a web traffic session for http communication by visiting `http://www.example.com`.
2. Captured the packet session using Wireshark and selected 6 packets for analyzing
3. Packets 29,33,35 shows the 3 way hand shake and 37,40,45
##### Packet 29 - SYN Packet
```
29	2.920961600	2401:4900:8fdd:c982:5b:8c30:57ce:518f	2606:4700:83b5:72db:f2fd:8fd:ef6b:ff98	TCP	86	18213 → 80 [SYN] Seq=0 Win=64800 Len=0 MSS=1440 WS=256 SACK_PERM

```
-- Source: My PC (`2401:4900:...:518f`) → Destination: example.com server (`2606:4700:...:ff98`)
- Layer 4: TCP, Src Port 18213 → Dst Port **80**
- Flags: **SYN** — initiating connection request
- Seq=0, Win=64800, MSS=1440, WS=256, SACK_PERM
- Purpose: _"My machine is requesting to establish a TCP connection with the web server"_




**Packet 33 — SYN, ACK**
```
33	2.992425000	2606:4700:83b5:72db:f2fd:8fd:ef6b:ff98	2401:4900:8fdd:c982:5b:8c30:57ce:518f	TCP	90	80 → 18213 [SYN, ACK] Seq=0 Ack=1 Win=65535 Len=0 MSS=1230 SACK_PERM WS=8192

```
- Source: example.com server → Destination: My PC
- Layer 4: TCP, Src Port 80 → Dst Port 18213
- Flags: **SYN, ACK** — server agrees to connection and acknowledges SYN
- Seq=0, Ack=1, MSS=1230
- Purpose: _"Server accepted connection request and acknowledged my SYN"_

**Packet 35 — ACK**
```
35	2.992530600	2401:4900:8fdd:c982:5b:8c30:57ce:518f	2606:4700:83b5:72db:f2fd:8fd:ef6b:ff98	TCP	74	18213 → 80 [ACK] Seq=1 Ack=1 Win=131584 Len=0

```

- Source: My PC → Destination: example.com server
- Layer 4: TCP, Src Port 18213 → Dst Port 80
- Flags: **ACK** — connection established
- Seq=1, Ack=1
- Purpose: _"3-way handshake complete — TCP connection established"_


**Packet 37 — HTTP GET Request**
```
37	2.993478100	2401:4900:8fdd:c982:5b:8c30:57ce:518f	2606:4700:83b5:72db:f2fd:8fd:ef6b:ff98	HTTP	427	GET / HTTP/1.1 

```
- My PC sends `GET / HTTP/1.1` to server
- Layer 7: HTTP request for root page `/`
- First application data after handshake
- `Host: www.example.com` header visible in plain text — confirms unencrypted request

**Packet 40 — ACK**
```
40	3.057554200	2606:4700:83b5:72db:f2fd:8fd:ef6b:ff98	2401:4900:8fdd:c982:5b:8c30:57ce:518f	TCP	78	80 → 18213 [ACK] Seq=1 Ack=354 Win=131072 Len=0

```
- Server acknowledges GET request
- Ack=354 — confirms all 353 bytes of GET request received

**Packet 45 — HTTP 200 OK**
```
45	3.063529800	2606:4700:83b5:72db:f2fd:8fd:ef6b:ff98	2401:4900:8fdd:c982:5b:8c30:57ce:518f	HTTP	83	HTTP/1.1 200 OK  (text/html)

```
- Server responds with `HTTP/1.1 200 OK (text/html)`
- Layer 7: Webpage content delivered in **plain readable text**
- Confirms unencrypted nature of HTTP


## Observation

_"Unlike HTTPS, HTTP traffic on port 80 is completely unencrypted. The full GET request, response headers, and HTML content were visible in plain text in Wireshark. This demonstrates why HTTP is considered insecure — any attacker on the same network could intercept and read this data. TCP connection was established via 3-way handshake before any HTTP data was exchanged, demonstrating how Layer 4 serves Layer 7."_
